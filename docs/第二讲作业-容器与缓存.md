# 第二讲作业：容器环境、负载均衡、动静分离、分布式缓存

## 1. 容器环境

### 1.1 Dockerfile
项目使用 Maven 构建的 Spring Boot 应用 Dockerfile：
```dockerfile
# ---- Build Stage ----
FROM maven:3.9.6-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline -B
COPY src ./src
RUN mvn clean package -DskipTests -B

# ---- Runtime Stage ----
FROM eclipse-temurin:17-jre-jammy
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
EXPOSE 8081
ENTRYPOINT ["java", "-jar", "app.jar"]
```

### 1.2 docker-compose.yml
使用 docker-compose 编排多个容器：
- **MySQL**: 数据库服务，端口 3306
- **Redis**: 缓存服务，端口 6379
- **seckill-app-1**: 后端实例1，端口 8081
- **seckill-app-2**: 后端实例2，端口 8082
- **Nginx**: 负载均衡器，端口 80

### 1.3 启动命令
```bash
docker-compose up -d
```

## 2. 负载均衡

### 2.1 Nginx 配置
Nginx 配置了多个负载均衡算法：

```nginx
# 算法1：轮询（Round Robin）- 默认
upstream seckill_backend {
    server seckill-app-1:8081;
    server seckill-app-2:8081;
}

# 算法2：加权轮询（Weighted Round Robin）
# upstream seckill_backend {
#     server seckill-app-1:8081 weight=3;
#     server seckill-app-2:8081 weight=1;
# }

# 算法3：IP Hash - 同一客户端IP始终路由到同一后端
# upstream seckill_backend {
#     ip_hash;
#     server seckill-app-1:8081;
#     server seckill-app-2:8081;
# }

# 算法4：最少连接（Least Connections）
# upstream seckill_backend {
#     least_conn;
#     server seckill-app-1:8081;
#     server seckill-app-2:8081;
# }
```

### 2.2 负载均衡算法说明

| 算法 | 说明 | 适用场景 |
|------|------|---------|
| 轮询 (Round Robin) | 按顺序依次分发请求 | 后端性能一致 |
| 加权轮询 | 按权重比例分发 | 后端性能不一致 |
| IP Hash | 同一IP路由到同一后端 | 有状态会话 |
| 最少连接 | 转发到当前连接最少的服务器 | 请求处理时间差异大 |

### 2.3 验证方法
使用 JMeter 进行压力测试：
1. 配置线程组模拟并发请求
2. 观察响应时间
3. 检查后端日志，验证请求分发

## 3. 动静分离

### 3.1 前端静态文件
前端项目位于 `frontend/html/seckill/`，包含：
- **HTML文件**: index.html, shop-list.html, shop-detail.html 等
- **CSS文件**: index.css, main.css, element.css 等
- **JS文件**: vue.js, axios.min.js, common.js 等
- **图片资源**: imgs/ 目录下各类图片

### 3.2 Nginx 动静分离配置
```nginx
server {
    listen 80;
    server_name localhost;

    # 静态资源 - 由 Nginx 直接返回
    location / {
        root   html/seckill;
        index  index.html;
        # 静态文件缓存
        expires 7d;
    }

    # API 请求 - 转发到后端
    location /api {  
        rewrite /api(/.*) $1 break;
        proxy_pass http://seckill_backend;
    }
}
```

### 3.3 性能对比
| 类型 | 响应时间 | 说明 |
|------|---------|------|
| 静态文件 | ~10ms | Nginx 直接返回 |
| 后端API | ~100ms | 需要业务处理 |

## 4. 分布式缓存

### 4.1 Redis 缓存实现
项目使用 Redis 实现商品详情页缓存：

**CacheClient.java** - 缓存工具类：
```java
// 缓存穿透：空值缓存
public <R,ID> R queryWithPathThrough(...) {
    // 1. 从Redis查询缓存
    // 2. 判断是否存在
    // 3. 不存在，查询数据库
    // 4. 将空值写入Redis防止缓存穿透
    // 5. 返回数据并存入缓存
}

// 逻辑过期解决缓存击穿
public <R,ID> R queryWithLogicExpire(...) {
    // 1. 查询缓存
    // 2. 判断是否过期
    // 3. 未过期直接返回
    // 4. 已过期则获取锁重建缓存
}
```

### 4.2 缓存问题解决方案

| 问题 | 解决方案 | 代码实现 |
|------|---------|---------|
| 缓存穿透 | 空值缓存 | `CacheClient.queryWithPathThrough()` |
| 缓存击穿 | 互斥锁/逻辑过期 | `ShopServiceImpl.queryWithMutex()` |
| 缓存雪崩 | 随机过期时间 | Redis Key 设置不同 TTL |

### 4.4 Redisson 分布式锁配置
```java
@Configuration
public class RedissonConfig {
    @Bean
    public RedissonClient redissonClient() {
        Config config = new Config();
        config.useSingleServer()
              .setAddress("redis://127.0.0.1:6379");
        config.setCodec(new org.redisson.client.codec.StringCodec());
        return Redisson.create(config);
    }
}
```
